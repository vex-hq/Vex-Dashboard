# Agent Readability for Vex Landing Site — Design

**Date:** 2026-04-25
**Status:** Approved (brainstorm complete, plan pending)
**Scope:** `apps/landing` only

---

## Problem

The Vex landing site is well-written for humans but invisible to AI agents that try to recommend, compare, or cite SaaS products. An audit against Neil Patel's AI-discoverability checklist found:

- Pricing page has zero `Product`/`Offer` JSON-LD despite carrying four priced plans inline.
- All seven `/compare/<vendor>` pages render comparison data as CSS grids with no schema and no semantic table markup.
- No `/api/*` routes exist — agents cannot fetch a machine-readable price or feature list.
- "Last updated" dates appear only on Privacy and Terms; pricing and comparison pages look stale.
- Stale "open source / Apache 2.0 / AGPLv3" claims still appear in JSON-LD, FAQ answers, and compare-page feature rows. Vex is no longer marketed as open source.

Without these signals an agent that *would* cite Vex has nothing structured to quote, and an agent that *should* cite Vex sees outdated facts.

## Out of Scope

- Distribution work (G2 listing, ProductHunt, HN, podcast outreach). Tracked separately as the larger lever; this design covers technical readiness only.
- `Review` / `AggregateRating` schema. Faking violates Google guidelines and AI policy. Add only after real reviews exist.
- `HowTo` schema on guides. High effort, low marginal value vs the existing `Article` + `BreadcrumbList`.
- Hero copy rewrites and full clarity pass. The brainstorm picked option B (mechanical + one positioning sentence), not C.
- MCP server, OpenAPI spec, `/api/compare`, `/api/faqs`. Speculative — no agent client expects them yet.

## Locked Decisions

| # | Decision |
|---|---|
| 1 | Technical-only fixes; distribution playbook tracked separately. |
| 2 | Mechanical changes plus a single positioning sentence on home + pricing. No other copy rewrites. |
| 3 | Agent-facing API surface = `/api/pricing` + `/llms.txt` only. |
| 4 | Positioning sentence: *"Vex helps founders shipping AI agents prevent hallucinations, drift, and policy violations from reaching customers through continuous runtime verification with 3-layer auto-correction."* |
| 5 | Vex is no longer open source. Scrub OSS claims across the entire landing site as part of this work. |
| 6 | Delivery shape = three sequenced PRs (schema + scrub → API surface → semantic HTML + dates + positioning). |

## Architecture

Single source of truth for every fact an agent might read.

```
apps/landing/
  lib/
    pricing.ts           # PLANS, LAST_UPDATED, CURRENCY (typed, readonly)
    site-meta.ts         # POSITIONING_SENTENCE, ORG, SAME_AS, FAQ entries
    seo/
      schemas.ts         # generators: productOfferSchema, breadcrumbSchema,
                         #             compareSchema(vendor), articleSchema(post)
      llms-txt.ts        # assembles /llms.txt body from pricing + site-meta + sitemap
  app/
    api/pricing/route.ts # GET → JSON of PLANS
    llms.txt/route.ts    # GET → text/plain
    pricing/page.tsx     # imports PLANS; renders Product+Offer JSON-LD
    page.tsx             # adds POSITIONING_SENTENCE to hero
    compare/<vendor>/page.tsx # renders compareSchema; OSS scrub
    guides/[framework]/page.tsx # adds BreadcrumbList + dateModified
    blog/[slug]/page.tsx        # adds BreadcrumbList + dateModified
    layout.tsx           # SoftwareApplication / Organization / FAQ from lib/seo;
                         # OSS scrub; expanded sameAs
  scripts/
    validate-schema.ts   # CI: fetches built pages, validates JSON-LD
```

**Principle:** every JSON-LD blob is generated by a function in `lib/seo/schemas.ts` from typed data in `lib/pricing.ts` and `lib/site-meta.ts`. No hand-edited JSON inline anywhere. `PLANS` is a `readonly` const — immutability respected.

**Payoff:** edit a price in one place → page, JSON-LD, `/api/pricing`, and `/llms.txt` all update consistently.

## PR Breakdown

### PR1 — Schema additions + OSS scrub *(highest leverage)*

**Data extraction.** Move the inline `plans` array out of `app/pricing/page.tsx` into `lib/pricing.ts` with a `Plan` interface and a `LAST_UPDATED` constant.

**Pricing JSON-LD.** Emit one `Product` with one `Offer` per plan (schema.org canonical pattern for SaaS variants):

```jsonc
{
  "@type": "Product",
  "name": "Vex",
  "description": "Runtime reliability layer for AI agents",
  "brand": { "@type": "Brand", "name": "Vex" },
  "offers": [
    {
      "@type": "Offer",
      "name": "Free", "price": "0", "priceCurrency": "USD",
      "category": "SaaS subscription",
      "eligibleCustomerType": "Sandbox for exploring agent reliability",
      "url": "https://app.tryvex.dev"
    }
    // Starter $29, Pro $99, Team $349 — each with priceSpecification
  ]
}
```

**Compare-page schema pattern.** `compareSchema(vendor)` emits:
- `SoftwareApplication` for Vex (`applicationCategory: "DeveloperApplication"`)
- `SoftwareApplication` for the competitor — factual fields only (name, URL). No invented ratings.
- `BreadcrumbList`: Home → Compare → `<vendor>`

Apply uniformly to all seven vendor pages.

**Guides + blog.** Add `BreadcrumbList` JSON-LD (currently visual only) and `dateModified` to existing `Article` schema.

**OSS scrub — exact changes:**

| File | Change |
|---|---|
| `app/layout.tsx` | Remove `"Open-source ..."` prefix in `SoftwareApplication.description`; remove `license` field; rewrite or drop FAQ "Is Vex open source?" entry |
| `app/pricing/page.tsx` | Drop FAQ "Is Vex open source?" |
| `app/compare/braintrust/page.tsx` | Remove `"Open source"` row from `features`; remove `(Apache 2.0)` from Vex description |
| `app/compare/{langfuse,langsmith,galileo,guardrails-ai,sentrial,concepts}/page.tsx` | Same scrub pattern; verify each before edit |

### PR2 — Agent API surface

**`app/api/pricing/route.ts`** — Next.js route handler with `force-static` so it prerenders into the build output:

```ts
import { NextResponse } from 'next/server';
import { PLANS, LAST_UPDATED, CURRENCY } from '~/lib/pricing';

export const dynamic = 'force-static';

export function GET() {
  return NextResponse.json(
    {
      product: 'Vex',
      currency: CURRENCY,
      lastUpdated: LAST_UPDATED,
      plans: PLANS,
      enterprise: { contact: 'hello@tryvex.dev' },
    },
    { headers: { 'Cache-Control': 'public, max-age=3600, s-maxage=86400' } },
  );
}
```

**`app/llms.txt/route.ts`** — plain-text following [llmstxt.org](https://llmstxt.org). Body assembled by `lib/seo/llms-txt.ts` from `PLANS`, `POSITIONING_SENTENCE`, and the sitemap function. Returned as `text/plain; charset=utf-8`.

```

> verification with 3-layer auto-correction.

Last updated: 2026-04-25
```

Linked from root layout `<head>` via `<link rel="alternate" type="text/plain" href="/llms.txt" />`.

### PR3 — Semantic HTML + dates + positioning sentence

**Pricing FAQ → `<details>`/`<summary>`.** Native disclosure, accessible by default, no JS, agent-friendly.

**Comparison tables → real `<table>` with `<caption>`, `<th scope>`.** Same Tailwind visual; agents and screen readers parse it as a table. Apply across all seven compare pages and the pricing-page `ComparisonTable` component.

**Pricing plan grid → `<ul role="list">` with `aria-labelledby` per plan card.**

**Visible "Last reviewed" line.** Small grey caption under H1 on pricing, every compare page, every guide. Pulled from the same `LAST_UPDATED` constant the JSON-LD reads — no drift between visible date and schema.

**Positioning sentence.** Stored in `lib/site-meta.ts` as `POSITIONING_SENTENCE`. Rendered:
- Below the existing hero `<p>` lede on `app/page.tsx`, above the CTA.
- Below the existing pricing lede on `app/pricing/page.tsx`.
- Reused in `/llms.txt`.

**Per-PR additions, all run in CI.** No quick-fix exceptions.

| PR3 | DOM (`@testing-library/react`): pricing FAQ renders as `<details>` with accessible `<summary>`; comparison table renders as `<table>` with `<caption>` and `<th scope>`. `axe-core` against `/pricing` and one `/compare/<vendor>` page — zero violations. Snapshot of `LAST_UPDATED` rendering. |

**Production validation per PR (manual, post-merge):**

2. Pricing page HTML through [Schema.org Validator](https://validator.schema.org/) — zero errors
3. [Google Rich Results Test](https://search.google.com/test/rich-results) — FAQ + Product eligible
4. `curl https://tryvex.dev/api/pricing | jq` — valid JSON, expected plans
5. `curl https://tryvex.dev/llms.txt` — renders as expected
6. `grep -ri "open source" .next/static` after build — no hits

## Error Handling

Static surface — failure modes are minimal by design.

- API routes use `force-static`. Failures surface at build time; CI fails loudly. Good.
- No try/catch around schema generation. Invalid data should crash the build, not silently ship stale schema.
- No fallbacks for missing `LAST_UPDATED` or `POSITIONING_SENTENCE` — they are `as const`, type-checked at compile time.

## Risks

| Risk | Mitigation |
|---|---|
| Schema validators reject our JSON-LD shape | PR1 snapshot tests + manual schema.org validator run pre-merge |
| OSS scrub misses a file and the forbidden-phrase guard blocks merge mid-PR | Run the guard locally first; iterate until clean before opening PR |
| `LAST_UPDATED` becomes stale because nobody updates it | Visible on pricing + compare pages, so any reviewer notices; bumping it is a one-line edit |
| `/llms.txt` convention changes before adoption | Body is auto-generated from a single function; format change is a single-file edit |
| Agents already cached the OSS-claim version | Time will fix it; new training cycles will pick up the corrected pages once dateModified bumps |

## Definition of Done

- All three PRs merged to `main` of the Dashboard repo.
- Six manual validation steps above pass on production.
- Forbidden-phrase guard green; no OSS claims survive in user-facing copy or schema.
- `/api/pricing` and `/llms.txt` reachable on tryvex.dev.
- `lib/pricing.ts` is the single source of truth — pricing page, JSON-LD, API, and llms.txt all read from it.
